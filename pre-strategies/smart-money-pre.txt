# PHASE 1: The Pine Script Foundation (White-Box)

According to **Doc 1**, a proper White-Box strategy avoids vague discretionary concepts and relies on a rigid **State Machine**, **Context Filters**, and explicit **Setup Detection**. 

For a 15m XAUUSD chart, Gold exhibits strong intraday seasonality (e.g., liquidity sweeps during the London/New York overlap). We will use **Strategy D (Structural / Price Action Codification)** mixed with **Component 2 (State Machine)**.

### The Strategy Idea: "Session Sweep & Structure Shift"
1. **Context Filter**: Only trade during the London/NY session overlap (high volume).
2. **Setup Detection**: Wait for price to sweep the Asian Session High or Low.
3. **Trigger**: After the sweep, look for a structural shift (e.g., a fast moving average crossing a slow one, or a candle closing back inside the range) indicating a mean-reversion or momentum injection.
4. **Risk Management**: ATR-based stop loss and fixed reward-to-risk (R:R) targets.

### Pine Script Architecture Outline
To codify this in Pine Script using the Canon's State Machine logic, your code should look conceptually like this:

```pine
//@version=5
strategy("XAUUSD 15m Structural Hybrid Base", overlay=true, initial_capital=10000, default_qty_type=strategy.percent_of_equity, default_qty_value=1)

// ==========================================
// COMPONENT 1: MARKET CONTEXT FILTER
// ==========================================
// Explicit, inspectable conditions (No learned weights)
in_session = not na(time(timeframe.period, "0800-1200:1234567", "America/New_York"))
daily_atr = ta.atr(14)
is_context_valid = in_session and daily_atr > 10 // Gold must have enough daily volatility

// ==========================================
// COMPONENT 3: SETUP DETECTION (Asian Range)
// ==========================================
var float asian_high = na
var float asian_low = na
if time(timeframe.period, "1800-0200:1234567", "America/New_York")
    asian_high := math.max(high, nz(asian_high, high))
    asian_low  := math.min(low, nz(asian_low, low))

// Sweep Detection
swept_high = high > asian_high and close < asian_high
swept_low  = low < asian_low and close > asian_low

// Momentum Trigger (Structure Shift proxy)
fast_ema = ta.ema(close, 9)
slow_ema = ta.ema(close, 21)
bullish_shift = ta.crossover(fast_ema, slow_ema)
bearish_shift = ta.crossunder(fast_ema, slow_ema)

// ==========================================
// COMPONENT 2: STATE MACHINE
// ==========================================
var string state = "IDLE" // States: IDLE, SETUP, IN_POSITION

if state == "IDLE" and is_context_valid
    if swept_low
        state := "SETUP_LONG"
    else if swept_high
        state := "SETUP_SHORT"

// ==========================================
// COMPONENT 4 & 5: ENTRY LOGIC & SIZING
// ==========================================
// Calculate risk per unit (ATR based)
stop_distance = ta.atr(14) * 1.5
target_distance = stop_distance * 2.0 // 2R Target

if state == "SETUP_LONG" and bullish_shift
    strategy.entry("Long", strategy.long)
    strategy.exit("Exit Long", "Long", stop=close - stop_distance, limit=close + target_distance)
    state := "IN_POSITION"

if state == "SETUP_SHORT" and bearish_shift
    strategy.entry("Short", strategy.short)
    strategy.exit("Exit Short", "Short", stop=close + stop_distance, limit=close - target_distance)
    state := "IN_POSITION"

// Reset State when flat
if strategy.position_size == 0
    state := "IDLE"
```

*Note: This White-Box Pine Script gives you baseline structural edge. However, it will suffer from "drawdowns in the wrong regime," which brings us to the Black-Box enhancements.*

# PHASE 2: Short-Term Black-Box Enhancements (Render / CPU-Only)

You want to improve this strategy without expensive GPUs, using a free server like Render (which offers ~512MB RAM and a basic CPU). Deep Learning (Neural Networks, LSTMs) is out of the question. 

Instead, we look to **Doc 1, Section 5.2: Hybrid Architecture Patterns**. The best approach is **Hybrid Pattern 2: White-Box Entry + ML Confirmation Filter** or **Hybrid Pattern 3: ML Regime Classification**.

### How to Build it (The Pipeline)
You will write a lightweight Python Flask/FastAPI application hosted on Render. TradingView will send a JSON webhook to Render whenever a setup occurs. Render will evaluate the data using a CPU-friendly Machine Learning model and return a "Go / No-Go" signal back to TradingView or directly to your broker.

### Strategy A: XGBoost/LightGBM Confirmation Filter (Tabular ML)
Gradient Boosted Trees (XGBoost, LightGBM) are the kings of tabular data and train in seconds on a standard laptop. They require almost zero computing power to run inference on Render.

1. **Feature Engineering Pipeline**: Download 5 years of 15m XAUUSD data. Write a Python script to compute features that your Pine Script doesn't see.
   * *Volatility Features*: `atr_5 / atr_20` (is volatility expanding?).
   * *Momentum Features*: RSI, MACD histogram slopes.
   * *Time Features*: Hour of day, Day of week.
   * *Target Label*: Did the "Session Sweep" trade hit 2R before hitting the 1.5 ATR stop loss? (1 = Yes, 0 = No).
2. **Train Offline**: Train a LightGBM classifier on your laptop. Save the model as a `.pkl` file.
3. **Deploy to Render**: Upload the `.pkl` to your Render web service.
4. **Execution**: When TradingView detects `state == "SETUP_LONG"`, it sends a webhook to Render with the current indicators. Render feeds this array to the `.pkl` model. If `probability of win > 0.60`, Render forwards the trade to your broker (e.g., via MetaTrader/Binance API).

### Strategy B: Hidden Markov Model (HMM) for Regime Classification
As Doc 1 states, HMMs are unsupervised probabilistic models that excel at detecting market regimes (Trending vs. Ranging, High-Vol vs. Low-Vol). They are purely statistical and computationally extremely cheap.
1. **The Model**: Use the `hmmlearn` Python library. Feed it the rolling 15m returns and rolling 15m volatility of Gold.
2. **The Output**: The HMM groups the market into 3 hidden states. You map these to reality:
   * State 0: Low-Vol Range (Asian Session usually)
   * State 1: High-Vol Trend (NY Open)
   * State 2: High-Vol Chop (News events)
3. **The Implementation**: Render continuously ingests Gold price data, updates the HMM, and hosts an API endpoint (`/current_regime`). Your execution script only takes the Pine Script trades if the HMM says the market is in "State 1" (Trend).

# PHASE 3: Long-Term Black-Box Enhancements (GPU / ML Infrastructure)

When you scale up and can afford GPUs, feature stores, and dedicated ML Ops pipelines, you transition from *filtering* human ideas to letting the machine *discover* the edge and manage the execution. 

Based on **Doc 1, Section 4 (Black-Box Strategies)**, here is what you can build:

### 1. Transformer/LSTM Sequence Models for Microstructure Pattern Prediction
Instead of explicitly defining a "Session Sweep" (which is inherently flawed by human visual biases), you feed raw data to a Deep Sequence Model.
* **The Concept (Strategy B: Pattern Prediction from Time Series)**: A Transformer model (like those used in LLMs, but adapted for time-series, e.g., Temporal Fusion Transformers) looks at the last 120 bars (30 hours) of XAUUSD OHLCV, plus the US Dollar Index (DXY) and US 10-Year Treasury Yields (US10Y) as cross-asset features.
* **Why GPU?**: Transformers process entire sequences simultaneously and require massive parallel matrix multiplications to learn "attention weights" (e.g., learning that a spike in US10Y yields 5 bars ago strongly correlates with a drop in Gold right now).
* **The Result**: The model outputs a continuous probability vector: *“There is a 72% probability Gold drops by 0.5% in the next 4 bars.”*

### 2. Reinforcement Learning (RL) for Trade Management
As noted in **Hybrid Pattern 4 (White-Box Signal + ML-Optimized Execution)** and **Component 7 (Trade Management)**, discretionary traders are terrible at trade management.
* **The Concept**: You keep the White-Box entry logic from TradingView. However, once in the trade, you hand the keys over to a Proximal Policy Optimization (PPO) Reinforcement Learning agent.
* **The Environment**: You simulate the Gold market. The agent's "State" is current PnL, time in trade, distance to average VWAP, and order book imbalance. Its "Actions" are: Hold, Move Stop to Breakeven, Take 50% Profit, or Close All.
* **Why GPU?**: RL requires millions of simulated episodes (trial and error) to learn an optimal policy. The agent will discover counter-intuitive management tactics, such as widening a stop loss temporarily during known liquidity hunts, then aggressively trailing it once the trend confirms.

### 3. NLP-Driven Macro Signal Generation (FinBERT)
Gold is a macro asset heavily driven by central bank policy and inflation data. Chart data alone is insufficient.
* **The Concept (Strategy D: NLP-Driven Signal Generation)**: Deploy an LLM/FinBERT model on a GPU instance. Connect it via WebSocket to a real-time news feed (e.g., Bloomberg, Reuters, or Federal Reserve press releases).
* **Execution**: The moment the FOMC rate decision is published, the GPU model reads the text, classifies it as "Hawkish/Bearish for Gold" with a magnitude score, and executes a market order in milliseconds—long before the 15m Pine Script chart even prints the next bar.

### Summary Roadmap
1. **Today (TradingView)**: Build the explicit White-Box structural setup (Sweep + Break of Structure + ATR sizing) in Pine Script. Ensure you have a strict State Machine.
2. **Next Week (Render/CPU)**: Export your backtest data. Train a LightGBM model offline to filter out the bad setups based on time/volatility features. Host the `.pkl` on Render to act as a gatekeeper for your TradingView webhooks.
3. **Next Year (GPU/Cloud)**: Move off TradingView. Build an end-to-end Python engine using Transformers to read raw price sequences and RL agents to dynamically trail stops based on order-flow data.
