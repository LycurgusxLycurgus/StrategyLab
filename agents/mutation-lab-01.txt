You are given an existing open-source trading strategy, usually in Pine Script, along with its stated market, timeframe, and logic. Extract the actual state logic from the code: regime filter, setup condition, entry trigger, stop logic, target logic, time stop logic, position sizing, cooldown logic, and any anti-reentry or anti-chop logic. Separate what is directly in the code from what is merely claimed in the script description.

Then evaluate the baseline through four gates in this order.

First, implementation integrity. Check for repaint risk, look-ahead bias, future leakage, hidden intrabar assumptions, unrealistic fills, ambiguous use of higher-timeframe data, and fake precision in stop/target handling. If integrity fails, stop there and classify the baseline as invalid until repaired.

Second, structural coherence. Ask whether the entry, stop, and target belong to one causal story. A strategy that mixes a trend entry with a mean-reversion exit, or a breakout entry with a chop filter that kills its edge, is structurally incoherent even if it looks sophisticated.

Third, sign of life. Evaluate trade count, expectancy, profit factor, average win versus average loss, and drawdown. Use the spirit of the lab thresholds rather than fake precision. A baseline survives only if it has a non-trivial sample and at least minimal economic life after realistic costs.

After the audit, make a hard routing decision. The valid outcomes are:
- kill now
- repair integrity first
- keep as baseline parent
- keep only as component
- run one controlled mutation
- bury

A controlled mutation means changing only one load-bearing feature at a time. Good examples are:
- one regime filter
- one entry confirmation
- one target redesign
- one stop redesign
- one side-specific restriction

Do not stack multiple new filters or redesign the whole strategy in one pass. If the baseline fails, prefer the smallest mutation that targets the main diagnosed failure mode.

1. Baseline Identity Contract

4. Structural Coherence Audit
5. Sign-of-Life Audit
6. Failure Localization
7. Gate Decision
8. One Controlled Mutation Only
9. What Must Not Be Changed Yet
10. Final Verdict

When you write the verdict, prefer disciplined prose over hype. The point of this workflow is to decide whether the baseline deserves to live before complexity is added.

1. Baseline Identity Contract

The narrowest defensible identity of this baseline is a BTC intraday trend-following system on the 15-minute chart, run in your screenshots on Bitstamp BTCUSD, with both long and short permissions. Mechanically, it is not a mean-reversion system. It is trying to capture trend continuation and trend flips through a dual moving-average structure, while using a crude anti-chop filter to avoid markets that are crossing the fast average too often. Its natural regime is not “all intraday conditions.” Its intended regime is persistent directional movement with relatively low local noise.

That matters because the way to judge it is different from the auction-value system. This script is supposed to win by being wrong often enough, but still making more on winners than it loses on losers. Your current metrics already show that logic: low win rate, but average win much larger than average loss.

2. Mechanical Reconstruction

The code calls the averages “EMA,” but it actually uses sma(close, 25) and sma(close, 100). The fast/slow relationship defines the local trend state. There are two long entry families and two short entry families. The aggressive long is a fast-over-slow crossover. The conservative long is a pullback entry where price is below the fast average, above the slow average, and the fast average is still above the slow average. Shorts are the exact mirror image.

The anti-noise logic is the script’s key regime filter. It counts how many times price crosses the fast average over the last 25 bars and only allows entries if that count stays below or equal to 10. So the script’s causal story is: trend or pullback entries are allowed only when the market has not been excessively ping-ponging around the fast average.

The exit logic is very simple. It computes ATR on a 15-minute resolution and sets a stop four ATRs away from the entry bar’s close. There is no take-profit, no partial, no breakeven rule, and no explicit time stop. Positions are therefore exited only by the stop or by reversal into the opposite side. Position sizing is not risk-based in percentage terms; in your current run it is effectively fixed-size, which is why the list of trades shows size 1.

So in plain language, the script is a trend/pullback entry engine with a volatility stop and no profit target. It wants to catch the part of the move after the trend structure is already visible, not the earliest breakout bar.

3. Integrity Audit

On pure implementation integrity, the script is not obviously broken. The security() call used for ATR in Pine v3 does not, by default, pull future data, because Pine v3 changed the default security behavior to lookahead_off. That means the current ATR call is not automatically a repaint trap just because it uses security().

The bigger integrity problem is not repainting. It is backtest realism. Your strategy declaration uses no commission and no slippage, and TradingView strategies are explicitly designed to simulate order behavior with those parameters available in the strategy framework. On a 15-minute intraday system that is trading 90 times in a short test window, leaving costs at zero is a load-bearing omission, not a cosmetic detail.

There is also a portability issue. Because the script is written in Pine v3, which TradingView itself labels as no longer actively maintained in its old documentation, and because the sizing is effectively fixed one-unit size, the current equity metrics are not yet a robust statement about how the system would behave across assets or even across BTC venues. That is not fatal, but it does mean the current result is a provisional research result rather than a production-grade baseline.

So integrity does not fail outright, but realism is incomplete.

4. Structural Coherence Audit

Structurally, this baseline is much more coherent than the value-area family you were iterating earlier.

The entries, stop, and holding logic belong to one causal story. The system assumes that once a directional move or pullback continuation starts in a relatively clean market, it should be given room to run. That is exactly what a wide ATR stop and no profit target are for. The low win rate is therefore not a contradiction. It is part of the design. Small frequent losses are the admission fee for catching the occasional large move.

The only real structural weakness is that the anti-chop filter is a little primitive. Counting price crosses of the fast average is a rough proxy for market quality, not a causal model of trend strength. But even that is not incoherent. It is just underpowered.

So this baseline passes the coherence test. It is a real trend-following conjecture, not a pile of mismatched parts.

5. Sign-of-Life Audit

This is the first baseline in this whole thread that shows genuine sign of life.

Within the tested March 15 to April 8, 2026 window shown in your screenshots, it produced 90 trades, a profit factor of 1.433, positive expected payoff, and a win/loss structure that makes sense for a trend-following strategy: only 38.89% winners, but average winners around 2.25 times the average losers. Max drawdown stayed around 2.64%, which is far cleaner than the v4 auction system. Both long and short sides are positive in the current test, with the short side somewhat stronger on profit factor. That is a real economic signal, not just cosmetic greenness.

At the same time, it is still too early to call this a pass. The window is short, the fees are zero, and intraday systems are especially vulnerable to cost drag. So the right reading is not “we found the final strategy.” The right reading is “this is the first parent candidate that actually deserves to keep living.”

6. Failure Localization

The main failure mode is not the entry idea itself. The main failure mode is regime detection plus cost sensitivity.

The metrics tell you exactly how this baseline works. It does not need a higher hit rate. Its economic engine is the gap between average win and average loss. So the levers that matter most are the ones that decide whether the script is trading in a market where runners are likely to develop. In this script, that load-bearing lever is the anti-chop filter, which currently lives in lookback_value and maxNoCross.

That also means the most important metrics are not buy-and-hold comparison or raw percent profitable. The metrics that actually matter here are trade count, cost-adjusted expected payoff, profit factor, average win versus average loss, and the difference in holding time between winners and losers. Your current run is telling you that winners last much longer than losers, which is exactly what you would expect from a decent trend-following baseline. That is why this script is worth keeping alive.

The levers that really move trade frequency are the fast and slow MA lengths, the crossover/pullback entry definitions, and especially the cross-count filter. The levers that really move quality are the same cross-count filter, ATR stop width, and later on any single trend-strength or session-quality filter. In this baseline, regime filtering matters more than fancy exits.

Keep as baseline parent.

I am not choosing “pass,” because the current backtest is still under-costed and too short-windowed.      max_labels_count = 500,
     max_lines_count = 500
)

// =========================
// Inputs
// =========================
fastLen         = input.int(25, "Fast SMA Length", minval = 1)
slowLen         = input.int(100, "Slow SMA Length", minval = 2)

lookbackValue   = input.int(25, "Noise Lookback", minval = 5)
maxNoCross      = input.int(10, "Max Fast-SMA Crosses", minval = 1)

atrLkb          = input.int(14, "ATR Stop Period", minval = 1)

// require the slow SMA to slope in the trade direction.

// =========================
// Aggressive entries: MA cross

That distinction is important, because your future system should not treat all mutations as equally plausible. It should prioritize local, story-consistent mutations first.

Your TradingView window is limited and likely regime-biased, so the current result is not “truth.” But it is strong enough to justify promotion into a deeper test environment.
