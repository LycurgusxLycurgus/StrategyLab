# SMART MONEY STRATEGY




Based on the directory structure and file names provided in the Google Drive link (`IPDA`, `Fractales`, `Liquidez`, `Orderblocks`, `FVG`, `Confirmaciones`, `KillZones`, `Fundamental`), this course teaches **Smart Money Concepts (SMC)**, heavily derived from the Inner Circle Trader (ICT) methodology. 

SMC is notoriously difficult to systematize because it masquerades as a strict rule-based system but relies heavily on human visual discretion and narrative-building. To translate this into a rigorous backtest engine in StrategyLab, we must strip away the subjective "hindsight" elements and isolate the structural mechanics.

Here is the exhaustive analysis of this strategy for implementation in StrategyLab.

---

### Edge Thesis
**The Core Belief:** Markets are not driven by retail buying/selling pressure, but by an algorithmic delivery of price (IPDA - Interbank Price Delivery Algorithm) that seeks to engineer liquidity (triggering stop losses) and rebalance pricing inefficiencies. 
**The Edge:** By waiting for specific time windows where institutional volume enters the market (KillZones), identifying where retail stops are pooled (Liquidity), and waiting for a false breakout (Sweep) followed by an aggressive reversal (Displacement) that leaves a gap in price (FVG), a trader can enter on the retracement alongside institutional order flow with a highly asymmetric risk-to-reward ratio.

---

### Likely Mistranslated Rules (State Variables)
When SMC is converted into a backtest engine, the strategy usually loses its edge because the following state variables are missing, over-constrained, or naively translated:

1. **The "Liquidity Pool" State (Inducement vs. Target)**
   * *The Mistranslation:* Backtests often define any recent swing high/low as a target or support/resistance.
   * *The Reality:* SMC treats un-swept swing points as *inducements* (traps to be broken) and already-swept points as *protected*. The engine is missing a state variable that tracks the **Mitigation Status** of structural pivots (e.g., `Untested`, `Swept`, `Engineered`).
2. **Multi-Timeframe Narrative (IPDA / Dealing Range State)**
   * *The Mistranslation:* Quants often translate "Higher Timeframe Bias" as a moving average regime (e.g., price > 200 SMA) or a simple RSI filter. 
   * *The Reality:* SMC bias is based on the current "Dealing Range" (the distance between the most recent HTF structural high and low). The missing state variable is the **Dealing Range Quadrant** (Premium vs. Discount), which dynamically updates only when a valid HTF structural break occurs, not on every bar.
3. **FVG (Fair Value Gap) Decay and Origin**
   * *The Mistranslation:* Executable rules often flag *any* 3-bar imbalance as an FVG and blindly place limit orders there.
   * *The Reality:* An FVG's validity decays over time and depends entirely on what caused it. An FVG is only high-probability if it was created by a liquidity sweep. The missing state variable is **FVG Provenance** (did this FVG break structure?) and **Mitigation Proportion** (has it already been filled past 50% / Consequent Encroachment?).
4. **KillZone Volatility Regime**
   * *The Mistranslation:* Coding KillZones purely as strict clock times (e.g., `If Time == 08:00 EST`).
   * *The Reality:* The edge of a KillZone relies on a volatility injection (often tied to the `Fundamental` module in the course). If the engine does not track an **Active Volatility State** (e.g., ATR expansion during the time window), it will take low-probability setups in dead markets.

### Parameters That Should Matter
To control behavior in a meaningful way, focus on these parameter families. Reject cosmetic parameters like "FVG Box Color", "Fibonacci extension levels" (which are just arbitrary take-profits), or "Orderblock lookback length".

* **Fractal Depth (Market Structure Sensitivity):** 
  * Controls what constitutes a structural high/low (e.g., a 3-bar fractal vs. a 5-bar fractal). This directly controls **trade frequency** and **setup quality**. A deeper fractal filters out micro-structure noise.
* **Sweep vs. BOS (Break of Structure) Threshold:**
  * Defines the boundary between a liquidity sweep (Judas Swing) and a genuine trend continuation. 
  * *Meaningful knob:* `Max_Close_Beyond_Pivot`. If price wicks beyond a pivot, it's a sweep. If the body closes beyond it by $X$ ATR, it's a BOS. This controls **regime selectivity**.
* **Displacement Minimum Threshold:**
  * Controls the required aggression of the reversal after a sweep. 
  * *Meaningful knob:* `Min_FVG_Size_ATR` (The FVG must be at least $X\%$ of the 14-period ATR). This filters out weak, low-quality setups.
* **Premium/Discount Execution Filter:**
  * *Meaningful knob:* `Dealing_Range_Threshold`. E.g., Only allow long entries if price is in the bottom 50% (Discount) of the current HTF dealing range.

### Minimum Executable Rule Set
To preserve the source material's edge, the execution logic must be structured as a strict sequence of events (a state machine).

**Mandatory Structural Conditions (The Setup Sequence):**
1. **Time Filter:** The current bar MUST be within a defined KillZone window (e.g., London or NY session).
2. **Liquidity Sweep:** Price MUST cross a valid unmitigated structural pivot (Liquidity Pool) and immediately reverse, closing back inside the range.
3. **Market Structure Shift (CHOCH):** Following the sweep, price MUST break a local structural pivot in the opposite direction of the sweep.
4. **Displacement (FVG Creation):** The leg that causes the CHOCH MUST contain a valid Fair Value Gap (a 3-bar sequence where the high of bar 1 and the low of bar 3 do not overlap).
5. **Entry Trigger:** A limit order is placed at the origin of the FVG or the Orderblock (the last opposite-colored candle before the displacement). The order is only valid for the duration of the current session.

**Optional Filters (For Regime Selectivity):**
1. **HTF Alignment:** The direction of the CHOCH must align with the Higher Timeframe Dealing Range bias.
2. **Premium/Discount Filter:** Long entries are only valid if the FVG is located below the 50% equilibrium of the recent swing leg.
3. **News Embargo:** Cancel all pending limit orders 15 minutes before Tier 1 macroeconomic data releases.

**Optimization Knobs:**
1. `Fractal_Lookback` (Integer: 3, 5, 7)
2. `KillZone_Start` / `KillZone_End` (Time)
3. `Min_Displacement_ATR_Multiplier` (Float: 0.5 to 2.0)
4. `Stop_Loss_Padding_ATR` (Float: 0.1 to 0.5) - How far below the Orderblock to place the hard stop.

### Recommended Representation in StrategyLab

Given the source material and the app's current architecture, this strategy is best represented as a **Hybrid Coded Family with LLM-assisted parameter evolution**.

**Why not pure Schema-Defined (JSON)?**
JSON cannot define the complex state machine required for SMC. You cannot declaratively express "track an array of unmitigated 3-bar gaps, wait for price to enter the top 50% of the gap, check if this gap was responsible for breaking a dynamically updating 5-bar fractal, and invalidate the gap if a candle body closes beyond it." That requires programmatic loops, array management, and custom object states.

**Why a Hybrid approach?**
SMC is highly structural (White-Box), but human traders apply immense subconscious statistical filtering to avoid bad setups. A pure White-Box coded engine will likely over-trade and fail because it lacks human intuition. 

By using a **Hybrid** approach, you establish a strict boundary:
* **The Engine Code (Python):** Handles the heavy lifting of the state machine. It calculates the Fractals, maintains the array of active FVGs, detects the sequence of Sweep $\rightarrow$ Displacement $\rightarrow$ CHOCH, and manages the order lifecycle.
* **The Declarative Layer (JSON / StrategyLab Manifest):** Controls the thresholds (`Min_Displacement_ATR`, `Fractal_Lookback`, `KillZone_Times`).
* **The LLM / Statistical Layer:** Once the White-Box engine runs across the parameter grid, you will generate a graveyard of failed trades. You can then feed the metadata of winners vs. losers to the LLM to discover the *statistical* edge that SMC traders call "intuition" (e.g., "The LLM noticed that setups fail 80% of the time if the FVG was created during the Asian session, propose a rule change to ignore Asian FVGs"). 

This preserves the interpretability of the Dalio/Macro perspective while allowing the Simons/Quant perspective to ruthlessly optimize the arbitrary thresholds that manual traders guess at.

A-ZERO STRATEGY

**For this strategy: A-Zero BTC Structural Range Liquidity Capture (SRLC)**

- `edge thesis`  
  BTC exhibits a repeatable structural pattern: liquidity pockets form at statistically recurring cycle extremes (rolling 60–90-day highs/lows), confirmed by clean bounces/rejections. Once locked, these create multi-week ranges where price mean-reverts to the opposite pocket with increasing directional conviction as the range matures (age-driven bias shift toward breakout). The edge is purely asset-structure driven — no events, no sentiment, no external data — producing positive expectancy through repeated liquidity-grab + range-fade + eventual breakout sequences.

- `likely mistranslated rules`  
  These 4 state variables are the highest-risk points where the strategy loses its real edge when turned into a backtest engine (missing, over-constrained, or incorrectly translated):  
  1. **Persistent rolling POI window state** — the 60–90-day min/max must be a true rolling buffer maintained bar-by-bar (e.g., deque or circular buffer). Most engines recompute naively or use fixed lookback arrays with lookahead leakage; this destroys macro-extreme detection fidelity.  
  2. **Range-active + regime-lock flag** — once a macro bottom/top is confirmed, the strategy must lock both the POI and regime (bullish/bearish) until invalidation. Without an explicit boolean + locked_price pair, the engine drifts into floating ranges or allows entries in undefined states.  
  3. **Stop-hunt recovery transient memory** — the “within 2 bars >50% recovery” relong rule requires a short-lived state (last_sl_hit_bar, recovery_threshold). Without it, backtests either ignore relongs or trigger them on any bounce, inflating win rate artificially.  
  4. **Brick-building sub-equity tracker** — realized TP1 profits must fund an independent parallel position with separate sizing. Simple total_equity sizing creates compounding errors and over-leverage; this state is almost always omitted in first-pass translations.

- `parameters that should matter`  
  Only families that directly modulate trade frequency, setup quality, or regime selectivity (rejecting cosmetic renames):  
  - POI lookback window (days): 60–90 → controls macro-extreme rarity and conviction (higher = fewer but higher-quality setups).  
  - Range width multiplier: 1.15–1.20 → defines zone size, TP distances, and frequency of full-range captures.  
  - Range maturity threshold (weeks/bars): 3–6 → core regime selector; flips from symmetric fade to breakout-biased behavior.  
  - Confirmation quality gate (RSI thresholds + bar count): RSI long <30–38, short >62–70 + 2–3 bar filter → prunes weak setups, directly raises edge per trade.  
  These five actually change the strategy thesis; everything else (exact buffer %, candle pattern wording, SL % if kept at 2%) is cosmetic.

- `minimum executable rule set`  
  Strict rules that must be preserved (distinguished by type):  

  **Mandatory structural conditions** (edge dies if removed or softened):  
  - POI = strict rolling 60–90-day extreme ±1% buffer.  
  - Extreme confirmation = exactly 2 consecutive closes ≥2% outside the POI.  
  - Range lock = Range Low/High fixed at confirmed POI; no entries allowed outside the locked bounds.  
  - Regime = bullish after bottom confirmation, bearish after top; shorts forbidden in bullish regime except pre-maturity fade.  
  - Invalidation = immediate flatten on any close >2% outside current range.  
  - SL = exactly 2% from entry (or POI extreme if tighter).  

  **Optional filters** (can be removed for baseline but improve Sharpe):  
  - RSI(14) + candle-type confirmation on 4H/1H.  
  - Age-based bias shift after maturity threshold.  
  - Stop-hunt relong condition.  

  **Optimization knobs** (tune only after structure is locked):  
  - POI lookback, width multiplier, maturity weeks, RSI levels, confirmation bar count.
